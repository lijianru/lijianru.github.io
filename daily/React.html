为什么选择使用框架而不是原生 JS？ ​

查看详情
  • 组件化：提高代码的复用性，降低维护的成本
  • 天然分层：原生代码涉及到页面 DOM 的更新大多耦合严重，现在的框架让代码分层和解耦，以 React 为例，如果需要改变 DOM 只需要更新数据即可
  • 生态：以 React 为例不管是数据管理，还是 UI 组件库都有比较成熟的解决方案
  • 开发效率：现代的框架默认自动更新 DOM，而非原生的手动操作，并对一些常见的问题进行了处理，比如 XSS

单页面应用和多页面应用的优缺点是什么 ​

查看详情
  • 单页面应用
    • 定义： 整个应用只有一个完整的页面。加载后，通过路由机制（如 React Router 或 Vue Router）动态重写当前页面，通过 AJAX/Fetch 获取数据。
    • 优点
      • 极致的体验： 页面切换无需重新加载整个 HTML，避免了“白屏”现象，提供类原生 App 的流畅感。
      • 前后端解耦： 后端只负责提供 API，前端负责逻辑与渲染，职责清晰。
      • 减轻服务器压力： 服务器只输出静态资源，大部分逻辑和渲染工作由客户端浏览器完成。
      • 资源缓存： 通用的 JS/CSS 只需加载一次，后续切换只按需请求数据。
    • 缺点
      • 首屏加载慢： 第一次打开需要下载大量的 JS 框架代码和业务包（可以通过路由懒加载缓解）。
      • SEO 不友好： 传统的爬虫不擅长执行 JS，很难抓取到由异步请求渲染的内容。
      • 内存占用高： 长时间运行不刷新页面，容易累积内存泄漏。
  • 多页面应用
    • 定义： 每次页面跳转，服务器都会返回一个新的 HTML 文档，浏览器重新加载整个资源。
    • 优点
      • SEO 友好： 每个页面都是独立的 HTML，内容直接写在源码里，极易被搜索引擎抓取。
      • 首屏加载快： 每个页面只加载当前所需的资源，不需要一次性下载庞大的 JS 运行环境。
      • 开发简单： 不需要复杂的路由管理和状态同步，适合小型站点。
    • 缺点
      • 体验生硬： 每次跳转都会触发页面刷新，用户会经历明显的白屏等待。
      • 重复加载： 相同的公共资源（Header/Footer/Libraries）在每个页面都会被重复解析和执行。
      • 前后端耦合： 逻辑通常混在服务器模板（如 JSP, Blade, EJS）中，维护成本高。

React 和其它框架的对比 ​

查看详情
  1. 核心哲学：你是哪种开发者？
  • React (灵活的乐高)： 它的本质是一个库。React 坚持 "Just JavaScript"（一切皆 JS），给予开发者极大的自由。你可以自由选择状态管理（Redux, Zustand）、路由（React Router, TanStack）等。
  • Vue (优雅的集大成者)： 它的定位是渐进式框架。它吸收了 React 的组件化和 Angular 的模板指令，形成了极其平衡的 DX（开发者体验）。它配套有官方维护的 Vue Router 和 Pinia，降低了选型心智负担。
  • Angular (严谨的现代化工厂)： 它是一个全功能平台 (Platform)。由 Google 维护，自带全套解决方案（HTTP, 路由, 表单, 注入）。它强调规范和工程化，非常适合大型企业团队协同。

  1. 技术特性对比表 | 特性 | React (19.x+) | Vue (3.5+) | Angular (20+) | | --- | --- | --- | --- | | 响应式原理 | VDOM + React Compiler (自动优化) | Proxy 细粒度响应式 (Vapor Mode) | Signals (信号) (走向无 Zone.js 时代) | | 语法风格 | JSX (逻辑与视图高度融合) | SFC (单文件组件: Template/Script/Style) | TypeScript + Decorators (类风格) | | 学习曲线 | 中等 (需理解 Hooks 与渲染闭包) | 平滑 (符合 HTML/CSS 直觉) | "陡峭 (需学习 RxJS, DI, TypeScript)" | | 性能表现 | 优秀 (编译器减少了手动 useMemo) | 极快 (编译后的按需更新) | 飞跃 (Signals 实现了局部极速更新) | | 主要背书 | Meta (Facebook) | 社区驱动 (尤雨溪) | Google |

  2. 2026 年的关键演进 作为面试者，我特别想提一下这三个框架在近两年的技术收敛和差异化：

  • React： 最大的变革是 React Compiler 的落地。过去开发者需要手动写 useMemo 和 useCallback 来防止重复渲染，现在编译器自动处理了这一切，大幅提升了开发效率和运行性能。
  • Vue： 随着 Vapor Mode (无虚拟 DOM 模式) 的成熟，Vue 进一步收割了对性能要求极高的边缘计算和嵌入式场景。它保持了“初学者友好”的同时，在高端性能上直追 Solid.js 等框架。
  • Angular： 正在经历“文艺复兴”。通过引入 Signals 和 Zoneless (去 Zone.js)，Angular 彻底解决了以往脏检查带来的性能损耗，代码变得更加现代和轻量，不再是那个沉重的“老人”了。

React 的特点 ​

查看详情
  1. 组件化
  • 可重用性：React 鼓励将用户界面分解成可重用的组件，每个组件封装了自己的结构、样式和行为。
  • 独立性：组件是独立的，便于维护和测试。
  • 嵌套和组合：组件可以嵌套和组合，构建复杂的用户界面。
  1. 声明式
  • 简化 UI 开发：通过声明式的编程方式，开发者只需描述 UI 应该呈现的状态，而不需要直接操作 DOM。
  • 可预测性：由于声明式编程，React 可以更容易地管理和调试 UI 状态变化。
  1. 单向数据流
  • 数据绑定：React 实现单向数据绑定，数据在组件树中自上而下流动。这种方式使得数据的流动和变化更加可预测和易于调试。
  • 状态管理：组件通过 props 接收数据，通过 state 管理内部状态。这种单向数据流确保了数据的变化是有规律和可跟踪的。
  1. JSX 语法
  • 增强可读性：JSX 是一种 JavaScript 的语法扩展，可以让你在 JavaScript 代码中直接编写类似 HTML 的模板。它提高了代码的可读性和开发效率。
  • 灵活性：JSX 最终会被编译成 React.createElement 调用，具有灵活的表达能力，可以轻松与 JavaScript 逻辑混合使用。
  1. Hooks
  • 函数组件中的状态和副作用管理：Hooks（如 useState、useEffect）允许在函数组件中管理状态和副作用，避免了使用类组件的复杂性。
  • 自定义 Hook：可以创建自定义 Hook 以重用逻辑代码，使代码更加简洁和可维护。

React 中循环渲染时 key 的作用 ​

查看详情
  1. 核心作用：给 DOM 节点一份“身份证” React 的 key 并不是传给组件的普通 props，它是 React 内部使用的唯一标识符。其核心作用是：
  • 识别节点的稳定性： 告诉 React 哪些元素在不同的渲染下保持不变、哪些被移动了、哪些被删除了。
  • 复用已有节点： 通过 key，React 可以高效地复用现有的 DOM 节点，而不是销毁并重新创建，从而极大地提升更新性能。
  1. 底层原理：Diff 算法的“捷径” React 在比较新旧虚拟 DOM 树时，默认采用的是同层级对比。
  • 如果没有 key： React 会按顺序简单粗暴地对比。例如列表头部插入了一个新元素，React 会认为所有的旧元素都改变了，从而导致整个列表的销毁与重建。
  • 有了 key： React 会维护一个 Map 结构，通过 key 直接找到旧树中对应的节点。
  • 如果 key 相同，React 只会更新属性（Patch），而不会重新挂载（Mount）。
  • 如果 key 发生了位置变化，React 只需要在 DOM 层执行 insertBefore 等移动操作，代价极小。
  1. 为什么不能用 Index 作为 Key？ 虽然用 index（索引）能消灭控制台警告，但在以下场景会产生严重问题：
  • 性能损耗： 如果你在列表头部插入数据，所有后续项的 index 都会改变。React 会误以为所有节点都更新了，导致大量的重复渲染。
  • 状态错位（最致命）： 如果列表项包含非受控组件或有自己的内部状态（如 useState），因为 React 认为“第一个位置还是第一个位置”，它会保留旧输入框里的文字，但数据却变成了新项的。这会导致严重的 UI 逻辑错误。
  1. 最佳实践建议
  • 唯一性与稳定性： 优先使用数据库返回的 ID（如 user.id）。
  • 避免随机数： 绝对不要用 Math.random() 作为 key。因为每次渲染 key 都会变，导致组件每次都被销毁重装，不仅性能极差，还会丢失输入焦点。
  • 万不得已用 Index： 只有当列表是静态的（不会排序、过滤、增删）且没有内部状态时，才考虑使用 index。

什么是真实DOM和虚拟 DOM ？它们的优点和缺点是什么？ ​

查看详情

真实DOM：DOM即为文档对象模型，页面最终渲染出的每一个节点都是一个真实的DOM。

  • 优点
    • 易用，无需借助任何框架
  • 缺点
    • 频繁操作DOM时，每次操作都会造成页面的重排和重绘
    • 代码可维护性差

虚拟 DOM： 本质上是一个 JavaScript 对象，包含 tag, props, children 三个属性。

  • 优点
    • 保证性能下限：虚拟 DOM 可以经过 diff 找出最小的差异，然后最终转化为真实的 DOM 展现在页面上，保证每次更新尽量小
    • 跨平台开发：虚拟 DOM 本质上是一个 JavaScript 对象，因此具有跨平台的能力
  • 缺点
    • 首次渲染大量的 DOM 时性能不如真实DOM，因为多了一个虚拟DOM的过程

React 16 的改进是什么？ ​

查看详情

React 16 是 React 历史上里程碑式的版本，它不仅仅是增加了一些 API，更是对底层架构进行了彻底的重写。

  1. 核心架构重写：React 16 引入了全新的架构——Fiber。在此之前（React 15），React 采用“栈调和（Stack Reconciliation）”，更新过程是同步且不可中断的，这会导致在处理大量计算时页面卡顿。
  2. 新特性
    • 错误处理 (Error Boundaries)
    • Fragments
    • Portal: 可将组件渲染到指定 DOM 树
    • 不再要求返回 React 元素，支持字符串、数字等作为组件的 render 返回值
    • Suspense(16.6)用于代码分割
    • hooks(16.8)

React 17 的改进是什么？ ​

查看详情
  • 事件委托的变更：在 React 17 中，React 将不再在document底层附加事件处理程序。相反，它会将它们附加到渲染 React 树的根 DOM 容器中。这是为了解决多个版本的react共存的问题。
  • JSX 转换改进：Babel 会自动调用编译器专用的新入口（如 _jsxRuntime），不再转换成 React.createElement。
    • 支持新的 JSX 转换（不再需要 import React from 'react'）
    • 配合 Babel 7.9+ 使用
  • 移除事件池：以前 React 为了性能，会重用事件对象。如果你在异步回调（如 setTimeout）里访问 e.target，会发现它变成了 null，必须手动调用 e.persist()。

React 18 的改进是什么？ ​

查看详情
  • Concurrent模式：并发渲染本身并不是一个功能。它是一个新的底层机制，使得 React 能够同时准备多个版本的 UI。并发模式的一个关键特性是渲染可中断，另一个例子是可重用状态。并发 React 可以从屏幕中移除部分 UI，然后在稍后将它们再添加回来，并重用之前的状态。并发模式解锁了许多的新功能，包括 Suspense，transition 和流式服务端渲染。在 React 18 中，使用 createRoot 来创建根节点，启用并发特性。
  • 自动批处理 (Automatic Batching)：在一个单独的重渲染事件中批量处理多个状态更新以此实现优化性能。在 React 18 之前，我们仅能够在 React 事件处理程序中批量更新。默认情况下 promise、setTimeout、原生应用的事件处理程序以及任何其他事件中的更新都不会被批量处理；但现在，这些更新内容都会被自动批处理。
  • 过渡更新：用于区分紧急和非紧急更新
    • 紧急更新：对应直接的交互，如输入，点击，按压等。
    • 过渡更新：将 UI 从一个视图过渡到另一个
    • 相关API：useTransition、startTransition
  • 新的 Suspense 特性：
    • 如果组件树的某个部分尚未准备好显示，Suspense 可以让你以声明的方式指定它的加载状态
    • 支持服务端 Suspense，并且使用并发渲染特性扩展了其功能（如果你在 Transition 期间挂起，React 不会让已显示的内容被后备方案取代。相反，React 会延迟渲染，直到有足够的数据，以防止出现加载状态错误）。
  • 新的客户端和服务端渲染APIs
    • createRoot
    • hydrateRoot
    • renderToPipeableStream
    • renderToReadableStream
  • 新的hooks：主要是提供给第三方库使用
    • useId：用于生成在客户端和服务端两侧都独一无二的 id
    • useTransition：useTransition 和 startTransition 让你能够将一些状态更新标记为过渡更新
    • useDeferredValue：允许推迟渲染树的非紧急更新

React 19 的改进是什么？ ​

查看详情
  • React Compiler 是一个 全新的编译器（又叫“函数组件编译器”），它的核心目标是：自动优化 React 应用性能，减少不必要的重新渲染 ——而且 不需要开发者手动写 memo、useMemo、useCallback。
  • Actions(使用异步过渡的函数): 让异步操作（如表单提交、状态变更）不再需要你手动操控 pending、错误处理与乐观更新。
    • useActionState： 自动管理异步请求的 pending（加载中）、data（返回数据）和 error（错误）状态。你不再需要为每个表单写三个 useState。
    • useFormStatus： 让嵌套在表单深处的子组件（如提交按钮）直接获取父表单的提交状态，无需通过 Props 层层传递。
    • useOptimistic： 官方支持乐观 UI 更新。在发送请求时立即假设成功并更新 UI，如果请求失败会自动回滚，非常适合点赞、评论等交互。
  • React 服务器组件：Server Components 是一种新的组件类型，它们会在打包之前、在独立于客户端应用或 SSR 服务器的环境中预先渲染。
  • 新的API use: 读取类似于 Promise 或 context 的资源的值。与 React Hook 不同的是，可以在循环和条件语句（如 if）中调用 use。但需要注意的是，调用 use 的函数仍然必须是一个组件或 Hook。
  • 新的 React DOM 静态 API
    • prerender
    • prerenderToNodeStream
  • 可以在函数组件中将 ref 作为 prop 进行访问
  • 可以将 <Context> 渲染为提供者，就无需再使用 <Context.Provider>
  • refs 支持清理函数

使用 Hooks 的优势是什么？ ​

查看详情
  • 类组件的不足
    • 状态逻辑难以复用：在组件间复用状态逻辑很难，可能用到 render props(渲染属性)或 HOC(高阶组件)，但都需要在原来的组件外层嵌套一层父容器，导致层级冗余
    • 趋向复杂难以维护：
      • 在生命周期中混杂不相干的逻辑
      • 类组件中到处都是对状态的访问和处理，导致组件难以拆分成更小的组件（更小的组件可以让组件的更新影响更小的范围，提高性能）
    • this 指向问题：父组件给子组件传递函数时必须绑定 this
  • Hooks 的优势
    • 解决了类组件的三大问题
    • 能在无需修改组件结构的情况下复用状态逻辑（自定义 Hooks）
    • 能将组件中相互关联的部分拆分成更小的函数（比如 ajax 请求）
    • 副作用的关注点分离：以前的副作用（ajax 请求，设置定时器等等）都写在生命周期函数中，而现在可以将所有的副作用拆分到各个 useEffect 函数中，一个 useEffect 只干一件事

简述 React 的 diff 算法，在有key和无key的时候diff算法是相同的吗？ ​

查看详情
  1. 在 React 中，Diff 算法被正式称为 Reconciliation（调和）。为了保证性能，React 做了三个前提假设（启发式算法）：
  • 只进行同层级比较（Tree Diff）。
  • 不同类型的组件会产生不同的树，直接销毁重建。
  • key 用于标识稳定节点。
  1. 无 Key 时的 Diff：原地复用（Patch）
  • 当列表没有 key 时，React 采用的是简单的顺序对比策略。
  • 逻辑： React 会按索引（Index）顺序一个个对比新旧 VDOM。
  • 操作： 只要组件类型（type）没变，React 就会认为这是同一个组件，直接原地更新（Patch）它的属性。
  1. 有 Key 时的 Diff：移动与复用（Map 映射） 当设置了 key，React 的 Diff 逻辑会变得更加智能。React 16+ 的 Fiber 架构下，针对多节点 Diff 主要分为两轮遍历：

  • 第一轮遍历：位置匹配

    • 按顺序对比。如果 key 和 type 都相同，则复用并继续。
    • 一旦发现 key 不匹配，立即中断第一轮遍历。

  • 第二轮遍历：处理移动、新增和删除

    • 建立 Map： React 会将剩余的旧子节点放入一个以 key 为键的 Map 中。
    • 查找复用： 遍历剩余的新节点，通过自己的 key 直接去 Map 中找。
    • 找到了： 标记为“移动”，复用该节点，只更新属性。
    • 没找到： 标记为“新增”。
    • 清理： 遍历结束后，Map 中剩下的旧节点标记为“删除”。

setState 什么时候是同步的，什么时候是异步的？ ​

查看详情
  • 在 React 中，如果是由 React 引发的事件处理（比如通过 onClick 引发的事件处理，在生命周期中调用 setState），调用 setState 不会同步更新 this.state，除此之外的 setState 调用会同步更新 this.state。所谓“除此之外”，指的是绕过 React 通过 addEventListener 直接添加的事件处理函数，还有通过 setTimeout/setInterval 产生的异步调用。
  • 这里所说的同步异步， 并不是真正的同步异步， 它还是同步执行的。这里的异步指的是多个 setState 会合成到一起进行批量更新。在 React 的 setState 函数实现中，会根据一个变量 isBatchingUpdates 判断是直接更新 this.state 还是放到队列中回头再说，而 isBatchingUpdates 默认是 false，也就表示 setState 会同步更新 this.state，但是，有一个函数 batchedUpdates，这个函数会把 isBatchingUpdates 修改为 true，而当 React 在调用事件处理函数之前就会调用这个 batchedUpdates，造成的后果，就是由 React 控制的事件处理过程 setState 不会同步更新 this.state。

在 React 类组件中，为什么修改状态要使用 setState 而不是用 this.state.xxx = xxx？ ​

查看详情

setState 方法做了很多关键的工作来确保组件能够正确地更新和重新渲染。直接修改 this.state 不会触发这些机制，从而导致应用行为不正确。

  • 触发重新渲染
  • 进行状态合并
  • 在setState后会进行声明周期方法的调用

简述 react 中的渲染和提交阶段 ​

查看详情

在 React 中，渲染（Render）和提交（Commit）是两个关键阶段，负责组件的更新和 UI 的渲染。

  • 渲染阶段是react确定更新内容的过程，在这个阶段react会创建或者更新fiber树，并计算需要改变的内容。这是一个纯计算的阶段，可以被中断。
  • 提交阶段是react将渲染阶段计算出的变化应用到真实DOM的过程，在这个阶段react会操作DOM，因此这个阶段是不可中断的。在此阶段会触发副作用函数，例如 useEffect、componentDidMount、componentDidUpdate 和 componentWillUnmount。

简述react的运行机制 ​

查看详情

我们写的JSX代码经过Babel编译后，最终会转换为React.createElement 函数（或新版的 JSX 运行时）的调用。当应用启动的时候，react会调用createRoot方法，并用根组件调用render方法递归得出所有的子组件，然后为它们创建DOM节点并使用appendChild方法将其添加到浏览器。

当组件的状态发生变化时，React 会为当前的组件节点创建一个更新任务，然后该任务会构建新的fiber树，然后进行diff，计算出需要更新的fiber节点，最终将变化的内容提交到浏览器。

在函数组件每一次函数上下文执行的时候，react 如何记录 hooks 的状态？ ​

查看详情

在 React 内部，Hooks 的状态并不是存储在函数组件内部的（因为函数执行完变量就销毁了），而是存储在与该组件对应的 Fiber 节点上。在 Fiber 节点中，有一个属性叫 memoizedState，它存储的是一个由 Hook 对象组成的单向链表。React 通过一个**全局指针（workInProgressHook）**来记录当前执行到了哪个 Hook。

多个 react hooks 如何记录每一个 hooks 的顺序的？/ 为什么不能条件语句中声明 hooks? ​

查看详情

每个 Hook 节点有一个next属性它将指向下一个hooks对象。然后每个hooks就以链表形式串联起来形成一个hooks链表。在条件语句中声明hooks，会破坏hooks的链表结构。

react 是怎么捕获到 hooks 的执行上下文，是在函数组件内部吗？ ​

查看详情

React 并不是在函数组件内部“捕获上下文”，而是依靠 Fiber + 内部全局指针（currentlyRenderingFiber 和 workInProgressHook）来跟踪 hooks 的执行上下文。函数组件每次执行时，React 会先设置好这些指针，然后 hook 调用顺序自然对应到 Fiber 上的状态。

为什么 useRef 不需要依赖注入就能访问到最新的改变值？ ​

查看详情

useRef 的本质是 React 在 Fiber 节点的 memoizedState 链表中，为你存放了一个普通 JavaScript 对象。其结构极其简单：

  • 地址唯一性： 在组件的整个生命周期内，React 保证每次渲染返回的都是同一个内存地址上的这个对象。
  • 修改不触发渲染： 修改 ref.current 只是修改了该对象的一个属性。在 JavaScript 中，这属于原地修改。

useEffect、useLayoutEffect和useInsertionEffect的区别是什么？ ​

查看详情
  1. 核心区别：触发时机 为了讲清楚区别，我们先看浏览器更新页面的标准顺序：
  • JavaScript 计算（React 运行 Render 阶段，生成 Fiber 树）。
  • React 提交（Commit）：
    • (A) useInsertionEffect 执行（DOM 变更前）。
    • (B) 修改真实 DOM。
    • (C) useLayoutEffect 执行（DOM 变更后，浏览器重绘前）。
    • 浏览器掉头进行布局（Layout）与绘制（Paint）。
    • (D) useEffect 执行（绘制完成后，异步执行）。
  1. 深度拆解

  • useEffect (最常用)

    • 特性： 异步执行。它不会阻塞浏览器的屏幕渲染。
    • 执行时机： 组件渲染并被绘制到屏幕之后。
    • 使用场景： 大多数副作用。如：数据请求、订阅事件、手动修改 DOM、记录日志等。
    • 优点： 性能最好，因为它让用户先看到 UI，再在后台处理逻辑。

  • useLayoutEffect (UI 修正)

    • 特性： 同步执行。它会阻塞浏览器的屏幕渲染。
    • 执行时机： DOM 更新之后，但在浏览器执行绘制（Paint）之前。
    • 使用场景： * 需要根据 DOM 元素的尺寸、位置来改变 UI。
    • 解决闪烁问题： 如果你在 useEffect 里修改样式，用户会先看到旧样式再看到新样式（闪烁）；在 useLayoutEffect 里修改，用户只会看到最终的结果。
    • 缺点： 如果逻辑太重，会导致页面白屏感或掉帧。

  • useInsertionEffect (CSS-in-JS 专属)

    • 特性： 同步执行。这是 React 18 专门为 CSS-in-JS 库（如 styled-components）设计的。
    • 执行时机： 在 DOM 变更之前。
    • 为什么要它： * 在 useLayoutEffect 或 useEffect 里插入 style 标签时，浏览器可能已经计算过一次样式了。
    • 插入新样式会导致浏览器重新计算所有元素的样式（Recalculate Style），这非常耗性能。
    • useInsertionEffect 确保在 DOM 变更前插入样式，让浏览器在后续 Layout 时一次性计算。
    • 注意： 业务开发中基本用不到，它主要给库作者使用。

react为什么没有使用浏览器提供的requestIdleCallback？ ​

查看详情
  1. 极不稳定的执行频率（浏览器差异）：requestIdleCallback 的触发频率非常不稳定。
  • 浏览器策略限制： 浏览器通常会优先保证动画和交互。如果页面有动画（如 requestAnimationFrame 正在运行），rIC 的触发频率会大幅降低。
  • 标签页切换： 当用户切换到后台标签页时，rIC 的触发频率可能会降到每秒一次甚至更低，这会导致 React 的更新逻辑在后台几乎停滞，影响用户切回页面时的体验。
  1. 刷新率限制 (FPS 不可控)：requestIdleCallback 的设计初衷是利用“空闲时间”，但它并不了解显示器的刷新率。
  • 现代显示器有 60Hz、90Hz 甚至 144Hz。
  • React 需要更精细地控制时间片 (Time Slicing)。自己实现调度器可以根据显示器的刷新率动态调整一帧的时间（通常目标是 16ms 或更短），而 rIC 往往无法提供这种高精度的控制。
  1. 优先级缺失 (Priority Management)：rIC 是一个“尽力而为”的 API，它没有内置的优先级概念。
  • React 的需求： React 需要区分离散交互（点击、输入 - 高优先级）、持续交互（滚动、拖拽 - 中优先级）和数据预取（低优先级）。
  • 自研调度器： React 的 Scheduler 内部实现了一套基于小顶堆 (Min-Heap) 的优先级队列。它能主动抢占低优先级任务，并确保高优先级任务（如用户正在输入）优先执行，这是原生的 rIC 无法做到的。
  1. 兼容性问题：在 React Fiber 研发初期，requestIdleCallback 的兼容性非常糟糕。
  • Safari 长期不支持： 直到很晚，Safari 才正式支持 rIC，这对于一个要运行在各种设备上的库来说是不可接受的。
  • IE 环境： 那个年代还需要考虑 IE，rIC 完全不存在。
  1. React 是如何替代它的？(MessageChannel)
  • 在早期的 Fiber 方案中，React 尝试过 requestAnimationFrame 配合 MessageChannel。
  • 在 React 18/19 的现代版本中，其核心逻辑大致如下：
  • 利用 MessageChannel： 这是一个宏任务（Macro Task）。
  • 时间片切分： React 会在当前帧开始时，计算一个截止时间（Deadline）（默认约 5ms）。
  • 主动让出： React 执行一小段任务，然后检查当前时间是否超过了截止时间。如果超时，它会通过 MessageChannel 发送一个消息，把执行权交还给浏览器（去处理渲染和事件），等浏览器处理完后，再回来继续执行刚才剩下的任务。

import React, { useState, useEffect, useRef } from 'react';

function Counter() {
  const [count, setCount] = useState(0);
  const prevCountRef = useRef();

  useEffect(() => {
    // 每次渲染完成后，更新 prevCountRef 的值
    prevCountRef.current = count;
  });

  const prevCount = prevCountRef.current;

  return (
    <div>
      <p>Current Count: {count}</p>
      <p>Previous Count: {prevCount}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

export default Counter;
